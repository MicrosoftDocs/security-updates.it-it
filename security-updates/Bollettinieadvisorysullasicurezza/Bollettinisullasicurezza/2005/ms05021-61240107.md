Security Bulletin

Bollettino Microsoft sulla sicurezza MS05-021 - Critico
=======================================================

Una vulnerabilità in Exchange Server può consentire l'esecuzione di codice in modalità remota (894549)
------------------------------------------------------------------------------------------------------

Data di pubblicazione: martedì 12 aprile 2005 | Aggiornamento: giovedì 14 aprile 2005

**Data di pubblicazione:** 12 aprile 2005
**Versione:** 1.0

#### Riepilogo

**Destinatari:** clienti che utilizzano Microsoft Exchange Server

**Effetti della vulnerabilità:** esecuzione di codice in modalità remota.

**Livello di gravità massimo:** critico

**Raccomandazioni:** è consigliabile applicare immediatamente l'aggiornamento.

**Aggiornamenti per la protezione sostituiti:** questo bollettino sostituisce un precedente aggiornamento per la protezione. Per un elenco completo, vedere la sezione Domande frequenti di questo bollettino.

**Avvertenze:** nessuna

**Software verificato e posizioni per il download dell'aggiornamento per la protezione:**

**Software interessato:**

-   Microsoft Exchange Server 2000 Service Pack 3 – [Download dell'aggiornamento](http://www.microsoft.com/downloads/details.aspx?familyid=2a2af17e-2e4a-4479-8ac9-b5544ea0bd66)
-   Microsoft Exchange Server 2003 – [Download dell'aggiornamento](http://www.microsoft.com/downloads/details.aspx?familyid=97f409eb-c8d0-4c94-a67b-5945e26c9267)
-   Microsoft Exchange Server 2003 Service Pack 1 – [Download dell'aggiornamento](http://www.microsoft.com/downloads/details.aspx?familyid=35bce74a-e84a-4035-bf18-196368f032cc)

**Software non interessato:**

-   Microsoft Exchange Server 5.5 Service Pack 4
-   Microsoft Exchange Server 5.0 Service Pack 2

I prodotti software inclusi nell'elenco sono stati sottoposti a test per determinare se tali versioni sono interessate dalla vulnerabilità. Le altre versioni non includono più il supporto degli aggiornamenti per la protezione o potrebbero non essere interessate dal problema. Per informazioni sulla disponibilità del supporto per il prodotto e la versione in uso, visitare il [sito Web di Microsoft Ciclo di vita del supporto](http://go.microsoft.com/fwlink/?linkid=21742).

Dettagli della vulnerabilità
----------------------------

<span></span>
#### Vulnerabilità in Exchange Server - CAN-2005-0560:

In Microsoft Exchange Server esiste una vulnerabilità legata all'esecuzione di codice in modalità remota che può essere sfruttata da un hacker per connettersi alla porta SMTP di un server Exchange ed eseguire un comando appositamente predisposto portando a una negazione del servizio o consentendo l'esecuzione di un programma dannoso nel contesto del servizio SMTP.

#### Fattori attenuanti della vulnerabilità in Exchange Server - CAN-2005-0560:

Exchange Server 2003 non elabora i comandi provenienti da utenti non autenticati.

Per prevenire questo tipo di attacco da Internet è possibile utilizzare Microsoft ISA Server 2000 o prodotti di terze parti che inoltrano e filtrano il traffico SMTP prima di inviarlo a Exchange. Informazioni dettagliate su come bloccare questo tipo di attacco utilizzando ISA Server sono reperibili all'indirizzo <http://www.microsoft.com/isaserver/support/prevent/how_to_block_can-2005-0560.doc>

I clienti che utilizzano ISA Server 2000 o ISA Server 2004 per la pubblicazione dei servizi SMTP di Exchange con le regole di pubblicazione SMTP predefinite sono meno esposti ad attacchi di questo tipo su Internet. Le regole di pubblicazione di ISA sono illustrate nella sezione "Soluzioni alternative".

#### Soluzioni alternative per la vulnerabilità in Exchange Server - CAN-2005-0560:

Microsoft ha verificato le soluzioni alternative seguenti. Sebbene queste soluzioni alternative non eliminino la vulnerabilità sottostante, consentono di bloccare gli attacchi noti. Se una soluzione alternativa riduce la funzionalità, viene identificata nella sezione seguente.

**Ispezione del protocollo SMTP per il filtraggio delle estensioni del protocollo SMTP.**

Le regole di pubblicazione predefinite di ISA per Exchange prevedono il filtraggio e l'eliminazione di tutte le estensioni del protocollo SMTP dal traffico che attraversa il firewall. Sono disponibili anche prodotti di terze parti che offrono funzionalità analoghe. Per ulteriori informazioni sulla pubblicazione di un computer Exchange Server con ISA Server, vedere l'[articolo della Microsoft Knowledge Base 311237](http://www.support.microsoft.com/kbid=311237).

**Accettazione delle sole sessioni SMTP autenticate.**

Se fattibile, accettare esclusivamente connessioni autenticate. L'accettazione delle sole sessioni provenienti da fonti attendibili impedirà ad hacker anonimi di sfruttare questa vulnerabilità.

Per richiedere l'autenticazione SMTP in un server Exchange 2000:

1.  Avviare Gestore di sistema di Exchange.
2.  Individuare il server nella struttura dell'organizzazione.
3.  Espandere il contenitore dei protocolli del server.
4.  Espandere il contenitore SMTP.
5.  Per ciascun server virtuale SMTP:

-   Aprire la finestra delle proprietà dell'oggetto server virtuale.
-   Fare clic sulla scheda Accesso.
-   Fare clic sul pulsante Autenticazione.
-   Deselezionare la casella di controllo "Accesso anonimo".
-   Scegliere OK per confermare le modifiche.

**Effetto della soluzione alternativa:**

in genere, i messaggi SMTP in ingresso vengono accettati senza richiedere l’autenticazione del mittente. Se l'utente adotta questa soluzione alternativa, potrà ricevere solo messaggi di posta elettronica dai mittenti ai quali sono state concesse le autorizzazioni necessarie nel sistema.

**Nota** Questa soluzione non impedisce a un utente malintenzionato che sia stato autenticato di sfruttare la vulnerabilità, ma protegge dagli attacchi sferrati da utenti anonimi.

**Blocco a livello di firewall della porta utilizzata da SMTP.**

Bloccare a livello di firewall la porta utilizzata da SMTP. In genere si tratta della porta 25.

**Effetto della soluzione alternativa:**

questa soluzione alternativa deve essere utilizzata come ultima risorsa per proteggersi dalla vulnerabilità. Potrebbe, infatti, incidere sulla possibilità di comunicare con sistemi esterni mediante posta elettronica.

**Deregistrazione di xlsasink.dll e ricorso ad Active Directory per la distribuzione delle informazioni di routing**.

1.  Nella directory bin dell'installazione di Exchange, eseguire regsvr32 /u xlsasink.dll.
2.  Se l'intervallo predefinito di un'ora necessario ai server Exchange per l'aggiornamento delle informazioni di routing da AD è sufficiente, passare al passaggio 8. In caso contrario, seguire le istruzioni riportate di seguito e tratte dalla sezione More Information dell'[articolo della Microsoft Knowledge Base 842026](http://support.microsoft.com/?kbid=842026).
3.  Eseguire regedit.
4.  Selezionare il percorso HKEY\_LOCAL\_MACHINE\\SYSTEM\\CurrentControlSet\\Services\\RESvc\\Parameters.
5.  Modificare il valore ReloadOsInterval. Nel caso in cui non esista, creare un nuovo valore DWORD con questo nome.
6.  Immettere il numero di secondi per l'intervallo di aggiornamento delle informazioni di routing da AD. L'impostazione predefinita è 3600.
7.  Fare clic su **OK** e chiudere regedit.
8.  Riavviare il server Exchange.

**Effetto della soluzione alternativa:**

I server Exchange non utilizzeranno SMTP per aggiornare attivamente le informazioni di routing. Se vengono apportate modifiche all'infrastruttura di posta elettronica, i server Exchange non rileveranno la nuova configurazione finché non eseguiranno l'aggiornamento delle informazioni di routing da Active Directory. Se l'impostazione dell'intervallo di aggiornamento è troppo elevata, ciò potrebbe causare un'interruzione temporanea dei servizi di posta elettronica.

#### Domande frequenti relative alla vulnerabilità in Exchange Server - CAN-2005-0560:

**Qual è la portata o l'impatto di questa vulnerabilità?**

Si tratta di una vulnerabilità legata all'esecuzione di codice in modalità remota. Se la vulnerabilità viene sfruttata, può consentire a un hacker di assumere in remoto il controllo completo del sistema interessato; l'hacker potrebbe quindi riuscire a installare programmi, visualizzare, modificare o eliminare dati oppure creare nuovi account con diritti completi.

**Quali sono le cause di questa vulnerabilità?**
Un buffer non controllato nel servizio SMTP.

**Che cos'è SMTP?**
SMTP (Simple Mail Transfer Protocol) è uno standard per il recapito dei messaggi di posta elettronica attraverso Internet, secondo quanto definito nei documenti [RFC 2821](http://www.ietf.org/rfc/rfc2821.txt?number=2821) e [RFC 2822](http://www.ietf.org/rfc/rfc2822.txt?number=2822). Il protocollo definisce il formato, i campi, il contenuto e le procedure di gestione dei messaggi di posta elettronica.

**Che cosa sono i verbi estesi SMTP?**
I verbi estesi SMTP sono definiti dal modello di estensione specificato nel documento [RFC 2821](http://www.ietf.org/rfc/rfc2821.txt?number=2821) e consentono l'aggiunta di nuove funzionalità al protocollo SMTP. Microsoft Exchange utilizza un verbo esteso per scambiare informazioni di routing e altre informazioni specifiche dell'applicazione tra i vari server Exchange presenti in un ambiente Exchange.

**A quali attacchi viene esposto il sistema a causa di questa vulnerabilità?**
Sfruttando questa vulnerabilità, un hacker potrebbe assumere il pieno controllo del sistema interessato.

**Quali capacità sono necessarie per sfruttare questa vulnerabilità?**
In Exchange 2000, la vulnerabilità può essere sfruttata da qualsiasi utente anonimo in grado di stabilire una connessione a una porta SMTP sul server Exchange e inviare una richiesta di verbi estesi appositamente predisposta.

In Exchange 2003, la vulnerabilità può essere sfruttata da qualsiasi utente autenticato in grado di stabilire una connessione a una porta SMTP sul server Exchange e utilizzare verbi estesi per inviare una richiesta appositamente predisposta.

**In quale modo un hacker può sfruttare questa vulnerabilità?**
È possibile sfruttare questa vulnerabilità connettendosi a una porta SMTP del server Exchange 2000 e inviando una richiesta di verbi estesi appositamente predisposta, permettendo all'autore dell'attacco di eseguire qualsiasi azione sul sistema, nel contesto di protezione del servizio SMTP. Per impostazione predefinita, il servizio SMTP viene eseguito con l'account LocalSystem.

Per sfruttare la vulnerabilità in Exchange 2003, l'hacker deve essere un utente autenticato nell'organizzazione di Exchange.

**Dal momento che Exchange 2000 Server utilizza il servizio SMTP di Windows 2000, questa vulnerabilità interessa anche il servizio SMTP di Windows 2000?**
No. La vulnerabilità non interessa il servizio Microsoft SMTP nei sistemi Windows 2000 in cui non è installato Exchange 2000 Server.

La vulnerabilità non interessa nemmeno i servizi SMTP che possono essere installati in Microsoft Windows NT® Server 4.0 o Windows XP.

**Questa vulnerabilità può essere sfruttata direttamente mediante la posta elettronica?**
No. Non è possibile sfruttare questa vulnerabilità inviando un messaggio di posta elettronica appositamente predisposto a una cassetta postale residente in un server Exchange. L'autore dell'attacco deve invece connettersi direttamente alla porta SMTP di un server Exchange.

**Quali sono gli scopi dell'aggiornamento?**
L'aggiornamento elimina la vulnerabilità modificando il modo in cui il servizio SMTP convalida la lunghezza di un messaggio prima di passarlo al buffer allocato.

L'aggiornamento per Exchange 2000 aggiunge requisiti di autenticazione analoghi a quelli già presenti in Exchange 2003.

**Al momento del rilascio di questo bollettino le informazioni sulla vulnerabilità erano disponibili pubblicamente?**
No. Microsoft ha ricevuto informazioni sulla vulnerabilità da fonti private.

**Systems Management Server:**

Microsoft Systems Management Server (SMS) offre una soluzione aziendale altamente configurabile per la gestione degli aggiornamenti. Tramite SMS gli amministratori possono identificare i sistemi Windows che richiedono gli aggiornamenti per la protezione ed eseguire il deployment controllato di tali aggiornamenti in tutta l'azienda, con il minimo disturbo per gli utenti finali. Per ulteriori informazioni sull'utilizzo di SMS 2003 per la distribuzione degli aggiornamenti per la protezione, visitare il [sito Web SMS 2003 Security Patch Management](http://go.microsoft.com/fwlink/?linkid=22939). Gli utenti di SMS 2.0 possono inoltre utilizzare [Software Updates Service Feature Pack](http://go.microsoft.com/fwlink/?linkid=33340) per semplificare il deployment degli aggiornamenti per la protezione. Per ulteriori informazioni su SMS, visitare il [sito Web SMS](http://www.microsoft.com/italy/smserver/default.mspx).

**Nota** SMS utilizza Microsoft Baseline Security Analyzer, lo strumento di rilevamento di Microsoft Office ed Enterprise Update Scanning Tool per offrire il più ampio supporto possibile per il rilevamento e il deployment degli aggiornamenti inclusi nei bollettini sulla sicurezza. Alcuni aggiornamenti non possono essere tuttavia rilevati tramite questi strumenti. In questi casi, per applicare gli aggiornamenti a computer specifici è possibile utilizzare le funzionalità di inventario di SMS. Per ulteriori informazioni sulla procedura da utilizzare, visitare il seguente [sito Web](http://go.microsoft.com/fwlink/?linkid=33341). Per alcuni aggiornamenti per la protezione può essere necessario disporre di diritti amministrativi ed eseguire il riavvio del sistema. Per installare tali aggiornamenti è possibile utilizzare lo strumento Elevated Rights Deployment Tool, disponibile in [SMS 2003 Administration Feature Pack](http://go.microsoft.com/fwlink/?linkid=33387) e in [SMS 2.0 Administration Feature Pack](http://go.microsoft.com/fwlink/?linkid=21161).
