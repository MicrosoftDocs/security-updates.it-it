Security Bulletin

Bollettino Microsoft sulla sicurezza MS12-006 - Importante
==========================================================

La vulnerabilità in SSL/TLS potrebbe consentire la divulgazione di informazioni (2643584)
-----------------------------------------------------------------------------------------

Data di pubblicazione: martedì 10 gennaio 2012 | Aggiornamento: mercoledì 17 luglio 2013

**Versione:** 1.3

### Informazioni generali

#### Riepilogo

Questo aggiornamento per la protezione risolve una vulnerabilità divulgata pubblicamente in SSL 3.0 e TLS 1.0. Questa vulnerabilità interessa il protocollo e non è specifica del sistema operativo Windows. Tale vulnerabilità può consentire la divulgazione di informazioni nel caso in cui un utente malintenzionato intercetti il traffico Web crittografato da un sistema interessato. TLS 1.1, TLS 1.2 e tutti i pacchetti di crittografia che non utilizzano la modalità CBC non sono interessati.

Questo aggiornamento per la protezione è considerato di livello importante per tutte le versioni supportate di Microsoft Windows. Per ulteriori informazioni, vedere la sottosezione **Software interessato e Software non interessato** in questa sezione.

L'aggiornamento per la protezione risolve la vulnerabilità modificando il modo in cui il componente Windows Secure Channel (SChannel) invia e riceve i pacchetti di rete crittografati. Per ulteriori informazioni sulla vulnerabilità, vedere la sottosezione Domande frequenti per la specifica voce della vulnerabilità nella sezione **Informazioni sulle vulnerabilità**.

Questo aggiornamento per la protezione risolve anche la vulnerabilità descritta per la prima volta nell'[Advisory Microsoft sulla sicurezza 2588513](http://technet.microsoft.com/it-it/security/advisory/2588513).

**Raccomandazione.** Se la funzionalità Aggiornamenti automatici è abilitata, gli utenti non devono intraprendere alcuna azione, poiché questo aggiornamento per la protezione viene scaricato e installato automaticamente. Gli utenti che non hanno attivato la funzionalità Aggiornamenti automatici devono verificare la disponibilità di aggiornamenti e installare questo aggiornamento manualmente. Per informazioni sulle opzioni di configurazione specifiche relative agli aggiornamenti automatici, vedere l'[articolo della Microsoft Knowledge Base 294871](http://support.microsoft.com/kb/294871).

Per gli amministratori e le installazioni delle organizzazioni o gli utenti finali che desiderano installare manualmente questo aggiornamento per la protezione, Microsoft consiglia di applicare l'aggiornamento quanto prima utilizzando il software di gestione degli aggiornamenti o verificando la disponibilità degli aggiornamenti tramite il servizio [Microsoft Update](http://www.update.microsoft.com/microsoftupdate/v6/vistadefault.aspx?ln=it-it).

Vedere anche la sezione, **Strumenti e informazioni sul rilevamento e sulla distribuzione**, riportata di seguito nel presente bollettino.

**Problemi noti.** L'[articolo della Microsoft Knowledge Base 2643584](http://support.microsoft.com/kb/2643584) descrive i problemi attualmente conosciuti che gli utenti potrebbero riscontrare durante l'installazione di questo aggiornamento per la protezione. L'articolo illustra inoltre le soluzioni consigliate in grado di risolvere questi problemi.

Domande frequenti sull'aggiornamento per la protezione  
------------------------------------------------------
  
<span></span>
**Dove si trova la sezione Informazioni sui file?**  
Fare riferimento alle tabelle contenute nella sezione **Distribuzione dell'aggiornamento per la protezione** per individuare la sezione Informazioni sui file.
  
**Qual è la relazione tra il presente aggiornamento per la protezione e il bollettino MS11-099?**  
Per essere protetti dal vettore di attacco dal Web attraverso Internet Explorer per la vulnerabilità dei protocolli SSL e TLS (CVE-2011-3389) come descritto in questo bollettino, i clienti devono installare entrambi questo aggiornamento MS12-006 e l'[Aggiornamento cumulativo per la protezione di Internet Explorer](http://go.microsoft.com/fwlink/?linkid=232505) MS11-099.
  
Sono necessari due aggiornamenti diversi poiché le modifiche richieste per risolvere il problema sono situate in prodotti Microsoft diversi. Questo aggiornamento, MS12-006, risolve la vulnerabilità che interessa WinHTTP e fornisce la possibilità di attivare la protezione in tutto il sistema. L'aggiornamento MS11-099 attiva queste protezioni per Internet Explorer.
  
**Perché questo bollettino contiene due aggiornamenti per i sistemi operativi** **Windows XP Professional x64 Edition Service Pack 2** **e** **Windows Server 2003?**  
Questo bollettino contiene due aggiornamenti, identificati dal numero KB del pacchetto, per i sistemi operativi Windows XP Professional x64 Edition Service Pack 2 e Windows Server 2003, come specificato nella tabella **Software interessato**. I due aggiornamenti sono necessari perché le modifiche richieste per risolvere la vulnerabilità nei sistemi operativi Windows XP Professional x64 Edition Service Pack 2 e Windows Server 2003 si trovano in componenti separati.
  
**Se si utilizza un sistema operativo** **Windows XP Professional x64 Edition Service Pack 2** **o** **Windows Server 2003, è necessario installare entrambi gli aggiornamenti offerti per il sistema?**  
Sì. Applicare tutti gli aggiornamenti offerti per il sistema operativo installato nei sistemi in uso.
  
**È necessario installare in una sequenza particolare questi aggiornamenti per la protezione?**  
No. È possibile installare più aggiornamenti per una versione di Microsoft Windows in un ordine qualsiasi.
  
**L'aggiornamento include modifiche di funzionalità relative alla sicurezza?**  
Sì. Per impostazione predefinita, questo aggiornamento viene attivato solo per le applicazioni che utilizzano il componente WinHTTP. Per gli sviluppatori che desiderano attivare questa funzionalità nella loro applicazione, consultare gli articoli di MSDN, [Funzione AcquireCredentialsHandle (Schannel)](http://msdn.microsoft.com/en-us/library/windows/desktop/aa374716(v=vs.85).aspx) e [Struttura SCHANNEL\_CRED](http://msdn.microsoft.com/en-us/library/windows/desktop/aa379810(v=vs.85).aspx).

Le altre versioni sono al termine del ciclo di vita del supporto. Vulnerabilità dei protocolli SSL e TLS - CVE-2011-3389  
------------------------------------------------------
  
<span></span>
Esiste una vulnerabilità legata all'intercettazione di informazioni personali nei protocolli di crittografia di SSL 3.0 e TLS 1.0. Questa vulnerabilità influenza il protocollo e non è specifica del sistema operativo Windows. Si tratta di una vulnerabilità legata all'intercettazione di informazioni personali che consente la decifratura di traffico SSL/TLS crittografato. Questa vulnerabilità principalmente si ripercuote sul traffico HTTPS poiché il browser è il vettore di attacco principale e interessa tutto il traffico Web servito tramite HTTPS o tramite il contenuto misto HTTP/HTTPS.
  
Per visualizzare questa vulnerabilità come voce standard nell'elenco CVE, vedere il codice [CVE-2011-3389](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3389).
  
#### Fattori attenuanti della vulnerabilità dei protocolli SSL e TLS - CVE-2011-3389
  
Per fattore attenuante si intende un'impostazione, una configurazione comune o una procedura consigliata generica esistente in uno stato predefinito in grado di ridurre la gravità nello sfruttamento di una vulnerabilità. I seguenti fattori attenuanti possono essere utili per l'utente:
  
-   TLS 1.1, TLS 1.2 e tutti i pacchetti di crittografia che non utilizzano la modalità CBC non sono interessati.
  
#### Soluzioni alternative per la vulnerabilità dei protocolli SSL e TLS - CVE-2011-3389
  
Per soluzione alternativa si intende un'impostazione o una modifica nella configurazione che non elimina la vulnerabilità sottostante, ma consente di bloccare gli attacchi noti prima di applicare l'aggiornamento. Microsoft ha verificato le seguenti soluzioni alternative e segnala nel corso della discussione se tali soluzioni riducono la funzionalità:
  
-   **Dare la priorità all'algoritmo RC4 nel software del server sui sistemi che eseguono Windows Vista, Windows Server 2008, Windows 7 o Windows Server 2008 R2**
  
    L'attacco interessa solo i pacchetti di crittografia che utilizzano gli algoritmi di crittografia simmetrici in modalità CBC, come AES e non interessa l'algoritmo RC4. È possibile dare la priorità all'algoritmo RC4 nel software del server per facilitare la comunicazione protetta utilizzando RC4 invece di CBC. Fare riferimento all'articolo MSDN, [Dare la priorità ai pacchetti di crittografia Schannel](http://msdn.microsoft.com/en-us/library/bb870930%28v=vs.85%29.aspx), per imparare a eseguire questa operazione.
  
    **Avvertenza** il client o il server con cui avviene la comunicazione deve supportare l'algoritmo RC4. Se il supporto per RC4 non è disponibile, viene utilizzato un pacchetto di crittografia diverso, se disponibile, e questa soluzione alternativa è inefficace.
  
-   **Attivare TLS 1.1 e/o 1.2 nel software client sui sistemi che eseguono Windows 7 o Windows Server 2008 R2**
  
    È possibile attivare una versione diversa del protocollo TLS che non è interessata da questa vulnerabilità. Per eseguire questa operazione è necessario modificare le Impostazioni di protezione avanzate di Internet Explorer.
  
    **Nota** Consultare l'[articolo della Microsoft Knowledge Base 2643584](http://support.microsoft.com/kb/2643584) al fine di utilizzare la soluzione **Microsoft Fix it** automatica per attivare o disattivare questa soluzione alternativa per TLS 1.1.
  
    **Metodo interattivo**
  
    Per modificare la versione di protocollo predefinita da utilizzare per le richieste HTTPS, eseguire i passaggi seguenti:
  
    1.  Dal menu **Strumenti** di Internet Explorer, scegliere **Opzioni Internet**.  
    2.  Nella finestra di dialogo **Opzioni Internet**, fare clic sulla scheda **Avanzate**.  
    3.  Nella categoria **Protezione**, selezionare le caselle di controllo **Utilizza TLS 1.1** e/o **Utilizza TLS 1.2**.  
    4.  Fare clic su **OK**.  
    5.  Chiudere e riavviare Internet Explorer.
  
    **Avvertenza** i server Web che non supportano TLS 1.1 o 1.2 eseguiranno la negoziazione SSL con versioni di SSL/TLS precedenti, annullando la soluzione alternativa.
  
    **Metodo di distribuzione gestito (attivare TLS 1.1 e disattivare SSLv2)**
  
    Salvare quanto segue in un file con estensione .REG, ad es. Disable\_SSLv2.reg:
  
    `Editor del Registro di sistema di Windows versione 5.00 [HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Internet Settings]"SecureProtocols"=dword:00000200 `
  
    Questo file .reg può essere applicato a singoli sistemi facendo doppio clic su di esso. Inoltre, può essere applicato in domini diversi utilizzando i Criteri di gruppo. Per ulteriori informazioni sui Criteri di gruppo, consultare l'articolo di TechNet [Insieme di Criteri di gruppo](http://go.microsoft.com/fwlink/?linkid=215719).
  
    **Avvertenza** i server Web che non supportano TLS 1.1 eseguiranno la negoziazione SSL con versioni di SSL/TLS precedenti, annullando la soluzione alternativa.
  
-   **Attivare TLS 1.1 nel software del server sui sistemi che eseguono Windows 7 o Windows Server 2008 R2**
  
    È possibile attivare TLS 1.1 che non è interessato dalla vulnerabilità.
  
    Consultare l'[articolo della Microsoft Knowledge Base 2643584](http://support.microsoft.com/kb/2643584) al fine di utilizzare la soluzione **Microsoft Fix it** automatica per attivare o disattivare questa soluzione alternativa per TLS 1.1.
  
    **Avvertenza** i client Web che non supportano TLS 1.1 eseguiranno la negoziazione SSL con versioni di SSL/TLS precedenti, annullando la soluzione alternativa.
  
    **Metodo di distribuzione gestita (attivare TLS 1.1)**
  
    Salvare quanto segue in un file con estensione .REG, ad es. Enable\_TLSv11.reg:
  
    `Editor del Registro di sistema di Windows versione 5.00 [HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\TLS 1.1\Client]"DisabledByDefault"=dword:00000000[HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\TLS 1.1\Server]"DisabledByDefault"=dword:00000000`
  
    Questo file .reg può essere applicato a singoli sistemi facendo doppio clic su di esso. Inoltre, può essere applicato in domini diversi utilizzando i Criteri di gruppo. Per ulteriori informazioni sui Criteri di gruppo, consultare l'articolo di TechNet [Insieme di Criteri di gruppo](http://go.microsoft.com/fwlink/?linkid=215719).
  
    **Avvertenza** i client Web che non supportano TLS 1.1 eseguiranno la negoziazione SSL con versioni di SSL/TLS precedenti, annullando la soluzione alternativa.
  
#### Domande frequenti sulla vulnerabilità dei protocolli SSL e TLS - CVE-2011-3389
  
**Qual è la portata o l'impatto di questa vulnerabilità?**  
Questa vulnerabilità riguarda l'intercettazione di informazioni personali. Un utente malintenzionato che ha sfruttato la vulnerabilità potrebbe decrittografare le parti di traffico crittografato e recuperare informazioni come i cookie di autenticazione.
  
**Quali sono le cause di questa vulnerabilità?**  
La vulnerabilità è causata da un difetto di progetto nei protocolli SSL e TLS, quando viene utilizzata la modalità operativa Cipher-Block Chaining (CBC).
  
**Che cosa sono i protocolli TLS e SSL?**  
Il protocollo di handshake di Transport Layer Security (TLS) è responsabile dell'autenticazione e dello scambio delle chiavi, vale a dire di due operazioni necessarie per stabilire o ripristinare delle sessioni protette. SSL (Secure Sockets Layer) è il predecessore del protocollo Transport Layer Security. Entrambi TLS e SSL eseguono le stesse funzioni e supportano comunicazioni di rete sicure mediante una combinazione di tecnologia pubblica e segreta principale.
  
Per ulteriori informazioni, vedere l'articolo di TechNet, [Funzionamento di TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).
  
**A quali attacchi viene esposto il sistema a causa di questa vulnerabilità?**  
Un utente malintenzionato in grado di sfruttare questa vulnerabilità che esegue il monitoraggio di una comunicazione crittografata può decrittografare il traffico crittografato intercettato.
  
**In che modo un utente malintenzionato può sfruttare questa** **vulnerabilità?**  
Un utente malintenzionato potrebbe iniettare codice dannoso in una risposta HTTP oppure utilizzare come host un sito Web appositamente predisposto contenente codice dannoso, forzando il browser ad eseguire tale codice. Questo codice invia diverse richieste, all'interno della stessa sessione TLS/SSL, a un sito Web HTTPS di terze parti nel quale i cookie vengono inviati automaticamente se è presente una sessione autenticata precedentemente. Questa è una condizione necessaria per sfruttare questa vulnerabilità. L'utente malintenzionato deve essere in grado di intercettare il traffico HTTP per sfruttare questa vulnerabilità in SSL, con il rischio di decrittografare porzioni di traffico crittografato (ad esempio i cookie di autenticazione).
  
**Quali sono i sistemi principalmente interessati da questa vulnerabilità?**  
Poiché il browser costituisce il vettore di attacco principale, tutti i sistemi, come le workstation o i terminal server, che inviano e ricevono traffico HTTP sono soggetti a questa vulnerabilità.
  
**Quali sono gli scopi dell'aggiornamento?**  
L'aggiornamento modifica il modo in cui il componente Windows Secure Channel (SChannel) invia e riceve i pacchetti di rete crittografati. Questo aggiornamento risolve la vulnerabilità che interessa WinHTTP e fornisce la possibilità di attivare la protezione in tutto il sistema. Tuttavia, per essere protetti dal vettore di attacco dal Web attraverso Internet Explorer per questa vulnerabilità, i clienti devono installare entrambi questo aggiornamento MS12-006 e l'[Aggiornamento cumulativo per la protezione di Internet Explorer](http://go.microsoft.com/fwlink/?linkid=232505) (2618444) MS11-099.
  
**Al momento del rilascio di questo bollettino, le informazioni sulla vulnerabilità erano disponibili pubblicamente?**  
Sì. Le informazioni sulla vulnerabilità sono state divulgate pubblicamente. Questa vulnerabilità è stata identificata con il codice [CVE-2011-3389](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3389).
  
**Al momento del rilascio di questo bollettino, erano già stati segnalati a Microsoft attacchi basati sullo sfruttamento di questa vulnerabilità?**  
No. Al momento della pubblicazione di questo bollettino Microsoft era a conoscenza dell'esistenza di codice di prova pubblicato, ma non aveva ricevuto alcuna segnalazione in merito allo sfruttamento pubblico di questa vulnerabilità a scopo di attacco.

Quando si installa questo aggiornamento per la protezione, il programma di installazione verifica se uno o più file da aggiornare nel sistema sono stati aggiornati in precedenza mediante una correzione rapida Microsoft.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**.
    3.  Nella casella **Nome del file o parte del nome** , digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

-   **Controllo delle chiavi del Registro di sistema**

    Per verificare i file installati dall'aggiornamento per la protezione, è anche possibile esaminare le chiavi del Registro di sistema elencate nella **Tabella di riferimento** in questa sezione.

    Queste chiavi del Registro di sistema potrebbero non contenere un elenco completo dei file installati. Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**.
    3.  Nella casella **Nome del file o parte del nome** , digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

-   **Controllo delle chiavi del Registro di sistema**

    Per verificare i file installati dall'aggiornamento per la protezione, è anche possibile esaminare le chiavi del Registro di sistema elencate nella **Tabella di riferimento** in questa sezione.

    Queste chiavi del Registro di sistema potrebbero non contenere un elenco completo dei file installati. Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.
