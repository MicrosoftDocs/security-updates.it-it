Security Bulletin

Bollettino Microsoft sulla sicurezza MS10-049 - Critico
=======================================================

Alcune vulnerabilità di SChannel possono consentire l'esecuzione di codice in modalità remota (980436)
------------------------------------------------------------------------------------------------------

Data di pubblicazione: martedì 10 agosto 2010 | Aggiornamento: mercoledì 1 settembre 2010

**Versione:** 1.1

### Informazioni generali

#### Riepilogo

Questo aggiornamento per la protezione risolve una vulnerabilità divulgata pubblicamente e una vulnerabilità segnalata privatamente del pacchetto di protezione Canale sicuro (SChannel) di Windows. La più grave di tali vulnerabilità può consentire l'esecuzione di codice in modalità remota se un utente visita un sito Web appositamente predisposto, progettato per sfruttare le vulnerabilità attraverso un browser Web Internet. In tutti questi casi, comunque, non è in alcun modo possibile obbligare gli utenti a visitare questi siti Web. L'utente malintenzionato dovrebbe invece convincere le vittime a visitare il sito, in genere inducendole a fare clic su un collegamento in un messaggio di posta elettronica o di Instant Messenger che le indirizzi al sito.

Questo aggiornamento per la protezione è considerato di livello critico per tutte le edizioni supportate di Microsoft Windows XP e Windows Server 2003 e di livello importante per tutte le edizioni supportate di Windows Vista, Windows Server 2008, Windows 7 e Windows Server 2008 R2. Per ulteriori informazioni, vedere la sottosezione **Software interessato e Software non interessato** in questa sezione.

L'aggiornamento per la protezione risolve le vulnerabilità implementando la RFC 5746 e una convalida aggiuntiva delle risposte SSL restituite da un server. Per ulteriori informazioni sulle vulnerabilità, cercare la voce specifica nella sottosezione Domande frequenti della sezione **Informazioni sulle vulnerabilità**.

Questo aggiornamento per la protezione risolve anche la vulnerabilità descritta nell'[Advisory Microsoft sulla sicurezza 977377](http://technet.microsoft.com/security/advisory/977377).

**Raccomandazione.** Se la funzionalità Aggiornamenti automatici è abilitata, gli utenti non devono intraprendere alcuna azione, poiché questo aggiornamento per la protezione viene scaricato e installato automaticamente. Gli utenti che non hanno attivato la funzionalità Aggiornamenti automatici devono verificare la disponibilità di aggiornamenti e installare questo aggiornamento manualmente. Per informazioni sulle opzioni di configurazione specifiche relative agli aggiornamenti automatici, vedere l'[articolo della Microsoft Knowledge Base 294871](http://support.microsoft.com/kb/294871).

Per gli amministratori e le installazioni delle organizzazioni o gli utenti finali che desiderano installare manualmente questo aggiornamento per la protezione, Microsoft consiglia di applicare immediatamente l'aggiornamento utilizzando il software di gestione degli aggiornamenti o verificando la disponibilità degli aggiornamenti tramite il servizio [Microsoft Update](http://www.update.microsoft.com/microsoftupdate/v6/vistadefault.aspx?ln=it-it).

Vedere anche la sezione, **Strumenti e informazioni sul rilevamento e sulla distribuzione**, riportata di seguito nel presente bollettino.

Vulnerabilità legata alla rinegoziazione TLS/SSL - CVE-2009-3555  
----------------------------------------------------------------
  
<span></span>
Una vulnerabilità legata ad attacchi di spoofing è stata riscontrata nel protocollo TLS/SSL, implementato nel componente di autenticazione di Microsoft Windows SChannel. Un utente malintenzionato in grado di sfruttare questa vulnerabilità può introdurre informazioni su una connessione TLS/SSL protetta, inviando efficacemente del traffico che provoca un attacco di spoofing contro il client autenticato. Questo aggiornamento risolve una vulnerabilità trattata precedentemente nell'[Advisory Microsoft sulla sicurezza 977377](http://technet.microsoft.com/security/advisory/977377).
  
Per visualizzare questa vulnerabilità come voce standard nell'elenco CVE, vedere il codice [CVE-2009-3555](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-3555).
  
#### Fattori attenuanti della vulnerabilità legata alla rinegoziazione TLS/SSL - CVE-2009-3555
  
Per fattore attenuante si intende un'impostazione, una configurazione comune o una procedura consigliata generica esistente in uno stato predefinito in grado di ridurre la gravità nello sfruttamento di una vulnerabilità. I seguenti fattori attenuanti possono essere utili per l'utente:
  
-   I siti Web che non ospitano contenuto via SSL, ma offrono esclusivamente contenuto attraverso connessioni HTTP (non crittografate) non sono interessati dalla vulnerabilità.  
-   Internet Information Services (IIS) 6 e IIS 7 non consentono la rinegoziazione avviata dal client. Ciò riduce in maniera significativa la superficie di attacco. È improbabile che i siti SSL in cui un utente digita nome utente e password siano esposti alla vulnerabilità. Lo scenario più probabile in cui potrebbe avere luogo un attacco di tipo spoofing con IIS 6 e IIS 7 è quello in cui i certificati sul lato client vengono utilizzati per eseguire l'autenticazione reciproca. Si tratta di uno scenario non comune.  
-   I clienti risultano interessati soltanto quando un utente malintenzionato è in grado di condurre un attacco di tipo "man-in-the-middle" sfruttando un'altra vulnerabilità, come un attacco alla subnet locale o uno spoofing del DNS.
  
#### Soluzioni alternative per la vulnerabilità legata alla rinegoziazione TLS/SSL - CVE-2009-3555
  
Per soluzione alternativa si intende un'impostazione o una modifica nella configurazione che non elimina la vulnerabilità sottostante, ma consente di bloccare gli attacchi noti prima di applicare l'aggiornamento. Microsoft ha verificato le seguenti soluzioni alternative e segnala nel corso della discussione se tali soluzioni riducono la funzionalità:
  
-   **Attivare SSLAlwaysNegoClientCert su IIS 6 e successiva**
  
    I server Web che eseguono IIS versione 6 e successiva e che sono interessati in quanto richiedono l'autenticazione reciproca assieme a un certificato client, possono essere rafforzati attivando l'impostazione SSLAlwaysNegoClientCert. Tale impostazione fa sì che IIS richieda al client un certificato in occasione della connessione iniziale senza bisogno di richiedere una rinegoziazione avviata dal server.
  
    **Impatto della soluzione alternativa:** L'impostazione di questo flag prevede che il client si autentichi prima che IIS possa caricare qualunque elemento dal sito Web protetto da SSL. Questa procedura fa sì che il browser richieda sempre all'utente un certificato client durante la connessione al sito Web protetto da SSL. Se *non* viene richiesta l'autenticazione reciproca per il contenuto protetto tramite SSL, gli utenti a cui viene richiesto di selezionare un certificato potranno semplicemente ignorare la finestra di dialogo con cui viene visualizzato il contenuto tramite la connessione SSL, senza dovere selezionare un certificato client con cui eseguire l'autenticazione.
  
    **Per IIS 6:**
  
    Eseguire il comando indicato di seguito dalla cartella "c:\\inetpub\\adminscripts" da un prompt dei comandi di amministratore con privilegi elevati:
  
    `adsutil.vbs SET w3svc/<N>/SSLAlwaysNegoClientCert true`
  
    in cui &lt;N&gt; rappresenta il numero del sito Web che deve essere configurato (il "sito Web predefinito" è 1, il sito Web successivo è 2 ecc.). Per proteggere il "sito Web predefinito" creato da IIS deve essere utilizzato ad esempio il comando indicato di seguito:
  
    `adsutil.vbs SET w3svc/1/SSLAlwaysNegoClientCert true`
  
    **Per IIS 7:**
  
    Salvare il testo indicato di seguito in un file chiamato "Enable\_SSL\_Renegotiate\_Workaround.js"
  
    `var vdirObj=GetObject("IIS://localhost/W3svc/1");// in questa riga sostituire 1 con il numero del sito Web che si desidera configurareWScript.Echo("Value of SSLAlwaysNegoClientCert Before: " + vdirObj.SSLAlwaysNegoClientCert);vdirObj.Put("SSLAlwaysNegoClientCert", true);vdirObj.SetInfo();WScript.Echo("Value of SSLAlwaysNegoClientCert After: " + vdirObj.SSLAlwaysNegoClientCert);`
  
    Eseguire il comando indicato di seguito da un prompt dei comandi di amministratore con privilegi elevati:
  
    `cscript.exe enable_ssl_renegotiate_workaround.js`
  
#### Domande frequenti sulla vulnerabilità legata alla rinegoziazione TLS/SSL - CVE-2009-3555
  
**Qual è la portata o l'impatto di questa vulnerabilità?**    
Si tratta di una vulnerabilità legata ad attacchi di spoofing. Un utente malintenzionato in grado di sfruttare questa vulnerabilità può introdurre informazioni su una connessione TLS/SSL protetta, inviando efficacemente del traffico che provoca un attacco di spoofing contro il client autenticato.
  
**Quali sono le cause di questa vulnerabilità?**    
Il protocollo TLS, che viene illustrato nel documento RFC 2246, descrive la funzione di rinegoziazione, che consente a entrambi i peer di rinegoziare in qualunque momento i parametri di una connessione protetta. Un utente malintenzionato in grado di trasformare un tipo di attacco come lo spoofing del DNS o un attacco alla subnet locale in un attacco di tipo "man-in-the-middle" all'interno di tale connessione, può sfruttare tale funzione di rinegoziazione per anteporre dei comandi specifici per l'applicazione a una sessione TLS valida in corso di configurazione.
  
In determinati scenari, il server potrebbe eseguire tali comandi nel contesto del client autenticato tramite TLS/SSL. In Windows i protocolli TLS e SSL vengono implementati dal pacchetto di protezione Canale sicuro (SChannel).
  
**Perché una installazione predefinita di IIS versione 6.0 e successiva non è interessata da questa vulnerabilità?**  
Questa vulnerabilità può essere sfruttata soltanto quando entrambi i peer di una connessione sono in grado di rinegoziare i parametri TLS/SSL. IIS impedisce qualunque rinegoziazione TLS/SSL avviata dal client. IIS avvia inoltre un tentativo di rinegoziazione sul lato server solamente quando è stato configurato in modo da utilizzare l'autenticazione reciproca basata su certificato, che non rappresenta una impostazione predefinita.
  
Notare che nelle distribuzioni in cui la connessione TLS/SSL effettiva non termina direttamente sul server IIS ma su un terminatore SSL di terze parti o su un proxy inverso, la connessione potrebbe essere esposta a questa vulnerabilità.
  
**Che cos'è un attacco di tipo "man-in-the-middle"?**  
Un attacco di tipo "man-in-the-middle" si verifica quando un utente malintenzionato riavvia la comunicazione tra due utenti utilizzando il proprio computer all'insaputa degli utenti che stanno comunicando. Ogni utente coinvolto nella comunicazione invia e riceve traffico dall'utente malintenzionato in modo inconsapevole, poiché crede di comunicare solamente con l'utente desiderato. Nel caso di questa vulnerabilità, il man-in-the-middle non è in grado di leggere, decrittografare o modificare la comunicazione crittografata che si svolge tra il client e il server. Il man-in-the-middle può solamente introdurre una richiesta nella sessione TLS che viene eseguita nel contesto del client.
  
**Che cos'è l'autenticazione reciproca?**  
L'autenticazione reciproca è una funzione di protezione in cui un processo client deve provare la propria identità a un server e il server deve provare la propria identità al client, prima che qualunque traffico di applicazione venga inviato attraverso la connessione client-server.
  
L'autenticazione reciproca prevede che il client e il server dimostrino le rispettive identità l'uno all'altro prima di eseguire qualunque funzione applicativa. L'identità può essere provata tramite strumenti crittografici, come una infrastruttura a chiave pubblica e i certificati digitali.
  
**Che cos'è TLS?**  
Il protocollo di handshake di Transport Layer Security (TLS) è responsabile dell'autenticazione e dello scambio delle chiavi, vale a dire di due operazioni necessarie per stabilire o ripristinare delle sessioni protette. Quando si stabilisce una sessione protetta, il protocollo di handshake gestisce le seguenti operazioni:
  
negoziazione di pacchetti di crittografia
  
autenticazione del server e facoltativamente del client
  
scambio di informazioni relative alla chiave di sessione
  
Per ulteriori informazioni, vedere l'articolo di TechNet, [Funzionamento di TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).
  
**Che cos'è SSL?**  
SSL è il predecessore del protocollo Transport Layer Security. Svolge le medesime funzioni e supporta comunicazioni di rete protette combinando una tecnologia di chiave pubblica con una tecnologia di chiave segreta.
  
Per ulteriori informazioni, vedere l'articolo di TechNet, [Funzionamento di TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).
  
**Che cos'è la rinegoziazione TLS/SSL?**  
La rinegoziazione è una funzionalità propria dei protocolli TLS e SSL che consente a entrambi i peer di rinegoziare dei parametri specifici per la connessione crittografica, come le chiavi di crittografia.
  
**Che cos'è il pacchetto di protezione SChannel?**    
Il pacchetto di protezione Canale sicuro (SChannel) è un SSP (Security Support Provider) che implementa i protocolli di autenticazione Internet standard SSL (Secure Sockets Layer) e TLS (Transport Layer Security). Questi componenti vengono utilizzati per garantire delle comunicazioni protette tra diverse applicazioni comuni in Internet e in rete, come ad esempio i browser Web. SChannel fa parte del pacchetto di protezione che fornisce un servizio di autenticazione in modo da garantire comunicazioni protette tra client e server.
  
Per ulteriori informazioni, vedere l'articolo di MSDN [Canale sicuro](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx).
  
**A quali attacchi viene esposto il sistema a causa di questa vulnerabilità?**    
Nelle distribuzioni vulnerabili un utente malintenzionato potrebbe utilizzare questa vulnerabilità al fine di anteporre del contenuto a una richiesta legittima avviata da un client nei confronti di un server nel contesto di una sessione valida autenticata tramite TLS/SSL. Questa vulnerabilità non consente all'utente malintenzionato di leggere, decrittografare o modificare il traffico crittografato che si svolge tra il client e il server.
  
**In che modo un utente malintenzionato può sfruttare questa vulnerabilità?**    
Un utente malintenzionato può sfruttare questa vulnerabilità preparando un attacco di tipo "man-in-the-middle" tra un client e un server, interrompendo la loro connessione non appena si verifica un tentativo di rinegoziazione SSL. A tal punto, l'utente malintenzionato può introdurre una serie limitata di dati sul percorso protetto, che il server interpreta come provenienti dal proprio client attendibile.
  
**Quali sono i sistemi principalmente interessati da questa vulnerabilità?**    
Questa vulnerabilità interessa principalmente i client utilizzati per i browser Web e i server Web corrispondenti, anche se alcune versioni di questo attacco possono interessare altri protocolli protetti TLS/SSL.
  
**Quali sono gli scopi dell'aggiornamento?**    
L'aggiornamento risolve la vulnerabilità implementando la RFC 5746. Questa RFC introduce un'estensione TLS che collega le rinegoziazioni TLS alla connessione TLS originale in cui sono eseguite, crittografandole. Ciò consente la progressione dell'autenticazione e la sua convalida ed impedisce l'attacco.

Sì. Le informazioni sulla vulnerabilità sono state divulgate pubblicamente. Questa vulnerabilità è stata identificata con il codice [CVE-2009-3555](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-3555).
  
**Al momento del rilascio di questo bollettino erano già stati segnalati attacchi basati sullo sfruttamento di questa vulnerabilità?**    
No. Al momento della pubblicazione del presente bollettino, Microsoft non aveva ricevuto alcuna segnalazione in merito allo sfruttamento di questa vulnerabilità a scopo di attacco.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**.
    3.  Nella casella **Nome del file o parte del nome**, digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

-   **Controllo delle chiavi del Registro di sistema**

    Per verificare i file installati dall'aggiornamento per la protezione, è anche possibile esaminare le chiavi del Registro di sistema elencate nella **Tabella di riferimento** in questa sezione.

    Queste chiavi del Registro di sistema potrebbero non contenere un elenco completo dei file installati. Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**.
    3.  Nella casella **Nome del file o parte del nome**, digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.

    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.
